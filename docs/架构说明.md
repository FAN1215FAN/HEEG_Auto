# 架构设计

## 当前主架构

项目当前按以下 5 层组织：

- 动作层：`src/heeg_auto/actions`
- 元素层：`src/heeg_auto/elements`
- 大模块层：`src/heeg_auto/modules`
- 用例层：`src/heeg_auto/cases`
- pytest 执行层：`tests` + `src/heeg_auto/runner`

## 各层职责

### 动作层

职责：

- 固定自然语言动作与代码实现映射
- 保证自然语言到代码的映射长期稳定

当前位置：

- `src/heeg_auto/actions/registry.py`
- `src/heeg_auto/core/actions.py`

### 元素层

职责：

- 固定每个大模块内部使用的元素清单
- 沉淀 AutomationId、标题、控件类型等定位信息

### 大模块层

职责：

- 定义一个完整可复用的业务模块
- 接收参数
- 内部组合动作与元素完成业务执行
- 通过命名断言组暴露标准化判定入口

### 用例层

职责：

- 定义完整测试场景
- 一个 case 里可以串多个大模块
- 用例必须有编号
- 可选定义 `变参`、`循环次数`、`失败即停`
- 通过 `断言组` 指定每个模块采用哪套业务判定逻辑

### 执行与治理层

职责：

- 收集 case
- 收集模块
- 展示 case 与模块链
- 展示变参/循环计划
- 执行真实 UI 套件
- `run_case_suite.py` 输出自定义 HTML 套件报告与单用例报告

## 当前执行流

1. `run_demo.py` 读取默认正式 case
2. `FormalCaseLoader` 把中文字段转成内部稳定字段，并规范化裸值参数
3. `FormalCaseRunner` 根据变参候选值和循环次数构建执行计划
4. `ModuleRunner` 读取大模块定义并执行步骤与指定断言组
5. `reporting.py` 输出单一 HTML 报告，并按轮次汇总 `PASS / FAIL / INTERRUPTED / NOT_RUN`
6. `run_case_suite.py` 直接执行所选 case，并生成套件级汇报报告
7. `pytest` 负责收集校验、治理测试和可选 UI smoke 联调

## 关键设计约束

- 正式用例优先中文字段，内部模块标识保持稳定 ASCII
- 一个 case 可以串多个大模块
- 一个 case 原则上只保留一组核心变参
- 所有多次执行场景统一遵循失败即停
- 大模块可编辑，底层执行逻辑稳定
- 断言组名称由 case 显式指定，具体判定标准由断言目录维护
- 动作、元素、模块与代码的对应关系必须文档化并持续更新
- 成功报告精简，故障报告展开
