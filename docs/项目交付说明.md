# 项目交付说明

## 1. 项目定位

HEEG Auto 是一个面向特定 WPF 上位机软件的自动化项目。它当前的目标不是搭建一个完全通用的低代码平台，而是固化一条可演示、可交付、可扩展、可维护的桌面 UI 自动化主线。

这条主线围绕以下能力建立：

- 用中文 YAML 表达正式业务 case
- 用稳定的动作映射承接自然语言动作
- 用元素清单沉淀控件定位资产
- 用可编辑模块 YAML 表达完整业务模块
- 用 runner 统一做 case 加载、参数解析、模块编排和执行
- 用 pytest 承担无人值守回归与治理校验，用 `run_case.py` 承接本地人工选择执行

## 2. 当前项目能解决什么问题

当前项目已经可以：

- 启动并连接目标 WPF 软件
- 以正式 case 的方式串接多个业务模块
- 执行“设备设置”“新建患者”等业务模块
- 在失败时自动保留截图、日志与自定义 HTML 报告
- 通过 YAML 维护业务参数，而不是频繁改底层 Python
- 为后续新增模块提供统一的元素、模块、case、pytest、文档落地方式

当前项目的边界也需要明确：

- 当前不是通用录制回放平台
- 当前不是任意项目都可直接复用的自动化引擎
- 当前主线优先验证“标准化 WPF 自动化工程”是否可持续扩展与交付
- 当前模块数量仍有限，但结构和治理链路已经成型

## 3. 整体结构

项目当前围绕 6 个核心层次组织：

### 3.1 `actions`

职责：维护自然语言动作与底层实现之间的固定映射。

例如：

- `单击` -> `click`
- `输入` -> `input_text`
- `下拉选择` -> `select_combo`
- `等待窗口` -> `wait_for_window`
- `断言窗口关闭` -> `assert_window_closed`

这一层的价值是把业务模块里的人类可读动作，稳定映射到代码实现，而不是在每个模块里直接写 Python 逻辑。

### 3.2 `elements`

职责：沉淀稳定控件定位信息。

典型内容包括：

- `automation_id`
- `control_type`
- `title`
- `page`
- `label`
- `aliases`

这一层只存控件本体与定位信息，不直接存业务流程。这样后续控件变化时，可以优先改元素层，而不是全仓库散改。

### 3.3 `modules`

职责：定义完整、可复用、可编辑的业务模块。

模块 YAML 中通常包含：

- `模块标识`
- `模块名称`
- `元素清单`
- `参数说明`
- `步骤`
- `断言`

这里描述的是完整业务过程，例如“打开设备设置 -> 设置参数 -> 点击确认 -> 断言关闭”，而不是孤立的一次点击。

### 3.4 `cases`

职责：定义正式业务用例。

当前正式 case 的核心结构是 `模块链`。一个 case 可以串多个模块，例如：

- `启动软件`
- `设备设置`
- `新建患者`

这样 case 层表达的就是完整业务场景，而不是只测试单个页面动作。

### 3.5 `runner`

职责：把 case、模块、元素和动作真正串起来执行。

主要承担：

- 加载正式 case YAML
- 将中文字段标准化为内部稳定字段
- 解析 `${变量}`
- 调度模块链
- 根据模块步骤构建 action payload
- 执行动作并汇总结果
- 失败时产出结构化结果和失败工件

### 3.6 `tests`

职责：以 pytest 为统一测试与治理入口。

这一层并不只是传统意义上的“写几个单元测试”，而是项目治理层的重要组成部分。

## 4. pytest 在当前项目中的具体作用

pytest 在当前项目中承担 5 类职责：

### 4.1 正式 case 收集与展示

`tests/support/case_catalog.py` 会扫描 `src/heeg_auto/cases/` 下的正式 case，并构造 pytest 可展示的测试项名称。

因此在 pytest 报告里，能看到类似：

- `公共功能_01 | 主流程 | 模块链: 启动软件 -> 新建患者 -> 设备设置`

这让 pytest 不只是“执行器”，同时还是正式 case 的统一展示层。

### 4.2 模块与结构治理

pytest 会校验：

- case 是否能被正确加载
- 模块注册是否完整
- 元素清单是否能正确解析
- loader / runner 的核心结构是否稳定

也就是说，pytest 是当前主线的治理层，而不仅是功能验证层。

### 4.3 真实 UI smoke 执行入口

`tests/smoke/test_patient_ui_flow.py` 会基于正式 case 驱动真实 UI 做治理联调。

因此 pytest 仍然是项目的治理与联调入口，但不再是日常正式 case 的唯一执行入口。

### 4.4 报告输出入口

当前项目区分两类报告：

- `run_case.py` / `run_demo.py` 输出自定义 HTML 报告，面向日常查看和汇报
- pytest 保留默认 HTML 治理报告，面向结构校验与联调记录

### 4.5 演示与批量执行的底层承接

当前 `run_demo.py` 和 `run_case.py` 都会直接调用 `FormalCaseRunner` 执行 case，不再通过 `pytest.main(...)` 间接承接。

这意味着：

- 对业务同事来说，日常入口会严格按 case 参数执行，不再额外追加非 case 指定的关软件/开软件动作
- 对工程治理来说，pytest 仍然负责校验、回归和可选 UI smoke

## 5. 自然语言与代码的对应关系

当前项目最关键的价值之一，是把自然语言配置和底层代码执行稳定连接起来。

### 5.1 对应关系

从上到下的关系是：

1. 正式 case 用中文 YAML 描述业务场景
2. case 中通过 `模块链` 调用业务模块
3. 模块 YAML 用中文动作描述步骤，例如 `单击`、`输入`、`等待窗口`
4. 动作注册表把中文动作映射到内部 action id
5. `ActionExecutor` 实现真正的 pywinauto 操作
6. 元素清单把中文业务控件名映射到稳定 locator
7. runner 在运行时把参数、元素、动作、断言组装成可执行链路

### 5.2 文字链路示意

可以用下面这条链路理解：

`正式 case YAML -> case_loader 标准化 -> module_runner 调度模块 -> ElementStore 解析元素 -> ActionExecutor 执行动作 -> pywinauto 驱动真实界面`

### 5.3 转化逻辑

#### case_loader 的作用

`src/heeg_auto/runner/case_loader.py` 负责：

- 读取中文正式 case
- 把中文字段转成内部稳定字段，如 `患者姓名 -> patient_name`
- 解析可选数据层和 `${timestamp}` 变量
- 把中文模块名映射为稳定模块 id

这一步解决的是“业务表达”和“内部执行字段”之间的转换问题。

#### module_runner 的作用

`src/heeg_auto/runner/模块执行器.py` 负责：

- 读取模块定义
- 根据参数构建上下文
- 解析步骤中的 `${参数}`
- 将元素引用转为真实 locator
- 调用动作执行器
- 汇总每个步骤的结果

这一步解决的是“模块 YAML”和“真正执行动作”之间的转换问题。

#### ActionExecutor 的作用

`src/heeg_auto/core/actions.py` 负责：

- 接收标准化 action payload
- 定位页面对象
- 触发点击、输入、下拉选择、窗口等待、断言等操作

这一步解决的是“动作定义”和“真实 UI 自动化代码”之间的转换问题。

## 6. 主要目录的作用

### 6.1 `src/heeg_auto/actions/`

维护动作注册表。这里定义自然语言动作与代码实现的固定关系。

### 6.2 `src/heeg_auto/elements/`

维护元素清单。按业务域组织稳定 locator 资产。

### 6.3 `src/heeg_auto/modules/`

维护业务模块 YAML。按业务域组织可编辑模块定义。

### 6.4 `src/heeg_auto/cases/`

维护正式 case。按业务域归档正式业务用例。

### 6.5 `src/heeg_auto/runner/`

维护 case 加载、模块调度、正式 case 执行链路。

### 6.6 `src/heeg_auto/core/`

维护底层驱动、页面基类、动作执行器、报告等底座能力。

### 6.7 `src/heeg_auto/pages/`

维护页面对象或弹窗页面辅助能力，承接复杂窗口/弹窗定位逻辑。

### 6.8 `src/heeg_auto/config/`

维护项目配置、超时、默认路径、定位兼容等底层配置。

### 6.9 `src/heeg_auto/legacy/`

归档历史原型与过渡实现，不再作为当前主线入口。

### 6.10 `tests/`

维护 pytest 治理层与 UI smoke 执行入口。

### 6.11 `artifacts/`

统一存放运行产物，例如：

- 日志
- 截图
- HTML 报告
- 其他失败工件

### 6.12 `docs/`

维护项目说明、运行说明、映射说明、维护规则和交付材料。

## 8. 当前项目的扩展方式

后续新增业务模块，建议继续按当前主线推进：

1. 整理控件信息
2. 生成元素清单 YAML
3. 生成业务模块 YAML
4. 注册模块
5. 补正式 case
6. 补 pytest 治理测试
7. 同步更新 mapping / control inventory / changelog 等文档

这条路径的价值在于：

- 每一层职责清晰
- 业务表达与执行实现分离
- 结构治理和真实执行可以同时演示
- 后续可继续扩展更多业务模块，而不需要推倒重来

## 9. 2026-04-01 当前阶段调整摘要

本轮调整后，正式 case 已新增以下统一规则：

- 顶层支持 `变参`、`循环次数`、`失败即停`
- 多轮执行按整条 case 展开，任一轮失败后后续轮次标记为 `NOT_RUN`
- 模块判定改为显式 `断言组`，不再使用 `预期状态: PASS/FAIL`
- 启动软件路径统一放入 `src/heeg_auto/config/app_config.yaml`
- 断言标准统一沉淀到 `src/heeg_auto/config/assertion_groups.yaml` 与 `docs/断言目录.md`
- 参数默认不加双引号；若值本身需要输入双引号，则用单引号包裹整段文本
