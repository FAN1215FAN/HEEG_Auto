# HEEG 自动化项目讲解报告

## 1. 项目定位

本项目是一个面向 `Neuracle.EEGRecorder.Viewer.HEEG.exe` 的 WPF 桌面自动化测试验证工程。当前阶段的目标不是直接做一个完整的低代码平台，而是先围绕一个真实业务场景验证自动化路线是否成立，并把底层能力沉淀为可复用的工程化资产。

本次选取的代表场景是“创建患者”。该场景覆盖按钮点击、文本输入、下拉选择、单选切换、结果断言、失败留证等多个关键能力，足以验证首版自动化链路。

## 2. 项目目标

本项目当前阶段主要验证以下问题：

- 是否能稳定驱动目标 WPF 软件完成真实业务操作
- 是否能将自动化能力按工程方式组织，而不是停留在一次性脚本层面
- 是否能在失败时自动保留有效证据，并输出统一报告
- 是否能开始向更低门槛的自动化表达方式演进

## 3. 为什么选择当前路线

### 3.1 为什么不直接做影刀式平台

如果项目一开始就直接做完整平台，会同时引入以下高风险问题：

- 目标软件 UI 自动化稳定性尚未验证
- 业务控件定位体系尚未沉淀
- 流程编排、可视化、拖拽交互开发成本高
- 一旦底层自动化不稳定，上层平台价值无法体现

因此本次选择“先验证底层能力，再考虑平台化”的路线：

1. 先把一个典型业务窗口做稳定
2. 先把控件定位、动作执行、失败取证和报告链路跑通
3. 先形成可扩展的自动化工程结构
4. 后续再逐步推进更低门槛表达和更高层编排能力

这条路线的核心优点是：控制复杂度、验证真实可行性、降低前期投入风险。

### 3.2 为什么选择 Python

本项目当前使用 Python，主要原因是：

- 与 Windows UIA 自动化库 `pywinauto` 配合成熟
- 原型验证速度快，适合快速试错
- 工程成本较低，适合首版落地
- 后续既可继续扩展自动化能力，也便于做辅助工具

## 4. 技术选型

当前项目使用的核心技术如下：

- 编程语言：Python
- 桌面自动化：`pywinauto` + UI Automation (`uia`)
- 用例表达：YAML + 中文行式脚本
- 测试执行：`pytest`
- 报告输出：HTML、JSON、Word
- 截图取证：Pillow / `ImageGrab`

整体技术选型围绕三个原则：

- 先稳定
- 先验证可落地
- 先为扩展预留结构

## 5. 项目整体思路

本项目没有把流程直接写成零散脚本，而是拆成几个层次：

- 配置层：应用路径、控件定位、默认用例配置
- 驱动层：负责启动应用、连接窗口、截图取证
- 动作层：统一封装点击、输入、选择、断言等通用动作
- 页面层：封装主界面和创建患者区域的页面操作
- 用例运行层：负责加载脚本、变量替换、顺序执行和结果收集
- 报告层：负责把执行结果生成 JSON / Word 报告

这样做的目的，是让这个项目从一开始就具备工程化和可扩展特征，而不是只做单次演示脚本。

## 7. 当前已经实现的能力

围绕“创建患者”场景，当前已经实现：

- 启动目标 exe 并识别主窗口
- 点击主界面“新增”按钮
- 等待“创建患者”区域出现
- 输入姓名、病历号、脑电号、备注等文本字段
- 选择性别、利手等控件
- 点击“确定”并验证结果
- 在失败时自动保留全屏、活动窗口、主窗口截图
- 每次运行后自动生成 JSON 报告和 Word 报告

## 8. 为什么同时保留 YAML 和中文行式脚本

这部分是本项目当前一个比较重要的设计取舍。

### YAML 的价值

- 结构稳定
- 更适合作为中间表示
- 更利于后续工具化生成

### 中文行式脚本的价值

- 更接近人工写步骤
- 对测试同事更友好
- 更适合低门槛自动化表达的验证

因此当前不是简单二选一，而是：

- 底层执行能力保持统一
- 上层同时验证结构化表达和低门槛表达


## 9. 当前边界与风险控制

为了保证首版可演示、可验证、可复用，当前明确控制了边界：

- 只先验证“创建患者”一个典型窗口
- 暂不处理复杂日期控件
- 暂不做大平台和拖拽编排
- 暂不扩展过多业务场景

这不是能力不足，而是阶段性控制复杂度，优先保证：

- 自动化链路真实可跑
- 失败证据完整
- 项目结构清晰
- 扩展路径明确

## 10. 后续扩展方向

后续如果继续推进，建议顺序如下：

1. 补充更多通用动作
2. 覆盖更多业务窗口
3. 增强更细粒度断言能力
4. 继续优化更低门槛脚本表达
5. 最后再考虑平台化、可视化、拖拽化

## 11. 常见质疑与回应

###  1：这个项目现在能证明什么

回应：
当前已经证明：

- 可以稳定驱动目标 WPF 软件完成真实业务动作
- 可以按工程化方式组织自动化代码
- 可以在失败时自动保留有效证据
- 可以自动生成结构化执行报告
- 可以向更低门槛表达方式演进

### 2：这个项目现在还不能证明什么

回应：
当前还不能直接证明完整平台化能力，也不能代表已经覆盖全部业务场景。但它已经完成了“核心路线可行、工程组织合理、扩展方向明确”的验证，这正是首阶段最重要的目标。

## 12. 结论建议

> 本次项目不是为了直接做一个完整平台，而是先用工程化方式验证 WPF 业务自动化的底层可行性；当前已经完成从真实业务执行、失败留证到自动报告的闭环，并为后续扩展更多业务场景和更低门槛的自动化表达方式打下基础。
