# 运行指南

## 环境准备

1. 在项目根目录创建虚拟环境：`python -m venv .venv`
2. 激活虚拟环境
3. 安装依赖：`pip install -r requirements.txt`

## 推荐运行方式

- 单次演示：`python run_demo.py`
- 执行测试：`python -m pytest`
- 导出控件树：`python run_inspector.py`
- 导出创建患者弹窗控件树：`python run_inspector.py --open-create-patient`

当前两个根目录入口脚本已经内置 `src` 路径引导，因此在 PyCharm 中直接运行 `run_demo.py` 或 `run_inspector.py` 时，不需要额外手动配置 `PYTHONPATH`。

## 中文行式脚本最小原型

`run_demo.py` 默认会执行 `src/heeg_auto/config/cases/create_patient.zh`。这种写法更接近测试同事直接写步骤：

```text
用例 create_patient_line_smoke
变量 patient_name autopatient${timestamp}
启动应用
单击 新增
等待窗口 创建患者
输入 姓名 ${patient_name}
单击 确定
断言窗口关闭 创建患者
断言文本可见 ${patient_name}
```

## 值的书写规则

这是当前最重要的一条规则：

- 直接输入固定文本：直接写文本，不要加 `${}`
- 引用变量：才使用 `${变量名}`

例如：

```text
输入 姓名 张三
输入 病历号 123456
输入 姓名 ${patient_name}
输入 脑电号 ${eeg_id}
```

下面这种写法会报错，因为它会被解释成“引用变量 `张三`”，而不是输入字面量：

```text
输入 姓名 ${张三}
```

## 失败截图策略

当前失败取证策略已经调整为：

- 运行异常时自动截图
- 断言失败时也自动截图
- 优先保存一张全屏截图，兼容单屏和扩展屏场景
- 同时尽量补一张当前活动窗口截图
- 如果主业务窗口可用，再补一张主窗口截图
- 文件名会带上用例名、失败步骤名和时间戳，方便回溯

## 自动报告策略

`run_demo.py` 每次执行后都会自动在 `artifacts/reports` 下生成一组同名报告文件，例如：

- `HEEG_Auto_Report_20260325_103000.json`
- `HEEG_Auto_Report_20260325_103000.docx`

当前报告约定：

- 报告语言统一为中文
- 一次运行生成一份报告
- Word 正文包含执行摘要、基本信息、步骤明细、失败信息和两张失败截图
- 成功截图暂不写入 Word，避免报告体量过大
- 详细错误堆栈和全部截图路径会保留在 Word 附录和 JSON 中

## 当前支持的中文行式命令

- `启动应用`
- `单击 <目标>`
- `输入 <目标> <值>`
- `下拉选择 <目标> <值>`
- `选择单选 <目标>`
- `等待窗口 <目标>`
- `等待可见 <目标>`
- `断言存在 <目标>`
- `断言窗口关闭 <目标>`
- `断言文本可见 <文本>`
- `截图 [文件名]`
- `变量 <键> <值>`
- `用例 <名称>`
- `说明 <描述>`

## 结构化 YAML 仍然保留

当前 YAML 用例依然可用，适合后续做模板化、生成式配置或更复杂的结构化表达：

`src/heeg_auto/config/cases/create_patient.yaml`

## 为什么同时保留两种形式

- 中文行式脚本更适合测试同事手写和快速搭建
- YAML 更适合作为结构化中间表示和后续工具化基础
- 当前最小原型的目的，是验证“低门槛表现形式 + 稳定底层引擎”这条路线是否成立
