# YAML 参数编写规范

本文用于统一正式 case YAML 的参数书写规则，避免不同人员写法不一致。

## 1. 总原则

- 默认所有参数不加双引号。
- case 层看到的值，默认都按“业务文本”理解。
- loader 会把数字、编号、IP 等裸值统一规范为文本再传给执行层。
- case 中不再写 `预期状态: PASS/FAIL`，改为显式 `断言组`。

## 2. 推荐写法

```yaml
患者姓名: 张三
性别: 女
病历号: 123
脑电号: 456
IP地址: 192.168.1.100
采样率: ${变参值}
断言组: 创建成功
```

## 3. 值中包含双引号时

如果软件里需要实际输入双引号字符，请使用单引号包裹整段文本，让双引号作为内容保留下来：

```yaml
病历号: '"123"'
备注: '患者姓名包含"双引号"'
```

## 4. 变量写法

- 变量引用默认不加双引号。
- 当前常见变量包括 `${timestamp}`、`${变参值}`。

例如：

```yaml
采样率: ${变参值}
患者姓名: autopatient${timestamp}
```

## 5. 变参与循环

推荐把核心变化项写在顶层：

```yaml
变参:
  模块: 设备设置
  参数: 采样率
  候选值:
    - 1000
    - 2000
    - 4000

循环次数: 2
失败即停: 是
```

说明：

- `候选值` 默认也不加双引号。
- 同一个 case 原则上只保留一组核心变参。
- 所有多次执行场景统一遵循失败即停。

## 6. 断言组写法

每个需要判定结果的模块都应显式声明 `断言组`：

```yaml
- 模块: 新建患者
  参数:
    患者姓名: 张三
    性别: 女
  断言组: 创建成功
```

## 7. 不推荐写法

以下写法都不推荐：

```yaml
病历号: "123"
采样率: "2000"
IP地址: "192.168.1.100"
预期状态: PASS
```

原因：

- 包裹双引号容易与“实际输入双引号”混淆。
- `预期状态` 无法明确告诉后续维护者实际采用了哪套判定逻辑。
