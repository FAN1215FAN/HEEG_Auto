# 维护流程

未来对本项目的任何修改，至少应遵守以下流程：

1. 修改前先阅读：
   - `说明.md`
   - `CHANGELOG.md`
   - `docs/项目交付说明.md`
   - `docs/需求说明.md`
   - `docs/架构说明.md`
   - `docs/目录说明.md`
   - `docs/运行指南.md`
   - `docs/YAML编写规范.md`
   - `docs/断言目录.md`
   - `docs/控件清单.md`
   - `docs/映射说明.md`
2. 优先沿当前主线结构修改：
   - `actions`
   - `elements`
   - `modules`
   - `cases`
   - `runner`
   - `tests`
   - `config`
3. 动作映射、元素清单、模块注册、断言目录和自然语言对应关系属于长期资产，不做随意跳层改动。
4. 修改代码后必须同步更新相关 Markdown 文档。
5. 重要结构、运行方式或交付口径变化时，至少同步更新：
   - `说明.md`
   - `docs/项目交付说明.md`
   - `docs/对话纪要.md`
   - `CHANGELOG.md`
6. 正式用例继续使用带编号的模块调用式 YAML，并显式声明 `断言组`。
7. 历史原型统一归档到 `src/heeg_auto/legacy/`，不要重新拉回主线目录。

## 扩展业务模块的固定步骤

未来新增业务模块时，建议按下面顺序推进：

1. 整理控件信息表
2. 补元素清单 YAML
3. 补业务模块 YAML
4. 补模块注册
5. 补 `src/heeg_auto/config/assertion_groups.yaml`
6. 补正式 case
7. 补 pytest 治理测试
8. 最后同步更新文档与清单

## 启动软件系统模块约定

- 启动软件作为显式系统模块写入 `模块链`
- 启动路径统一维护在 `src/heeg_auto/config/app_config.yaml`
- case 中不再直接写死软件路径
- 推荐参数：`会话模式`
- 业务 case 如需连续运行，可通过 `会话模式: 复用已有应用` 明确要求复用当前会话

## case 编写约定

- 一个 case 原则上只保留一组核心变参
- 所有多次执行场景统一遵循失败即停
- 默认所有参数不加双引号
- 如果值本身需要输入双引号，使用单引号包裹整段文本
- 通过/失败判定优先引用标准断言组，不再使用 `预期状态: PASS/FAIL`

## 下拉控件类模块约定

对于设备设置这类下拉较多的模块：

- 第一轮优先沉淀控件本体，而不是枚举全部下拉值
- 具体业务值默认写在 case 层参数中
- 纯展示字段在第一轮可以先不进入元素清单，除非用户明确要求断言它们
