# YAML 编写规范

## 文档定位

这份文档说明当前正式步骤式 case 的写法、默认规则、`数据` 与 `变参` 的用法，以及图形交互类 case 的补充约定。

## 一、正式 case 顶层结构

```yaml
用例编号:
用例名称:
标签:
数据:
变参:
循环次数:
失败即停:
步骤:
  - 名称:
    窗口:
    元素:
    按钮:
    动作:
    值:
    文本:
    超时:
    参数:
    断言:
    可选:
```

## 二、基本规则

1. `名称` 不是强制字段，但强烈建议保留。
2. 普通表单填写优先写成 `窗口 + 参数`。
3. 点击类交互优先写成 `窗口 + 按钮` 或 `窗口 + 元素`。
4. 复杂结果优先写命名断言，不直接把底层检查堆进 case。
5. 标签里不再使用版本号式标签。

## 三、默认动作规则

### 1. 写了 `按钮`

- 默认就是点击

### 2. 写了 `元素`，但没写 `动作`

- 默认按控件类型推断，通常是点击

### 3. 写了 `参数`，但没写 `动作 / 按钮 / 元素`

- 默认按表单填写处理

### 4. 显式动作只在必要时写

例如：

- 拖动
- 双击
- 右键
- 等待窗口
- 截图

## 四、`数据` 段职责

`数据` 段用于承载当前 case 中频繁改动的业务变量和上下文变量。

适合放在 `数据` 中的内容：

- 患者姓名、病历号、脑电号
- 设备类型、采样率
- 图形交互比例
- 剪辑开始时间、剪辑结束时间、主窗口新增记录时长

结论：

- 改 case 时优先改 `数据`
- 不要把同一类变量分散写到很多步骤里

## 五、`变参` 用法

`变参` 只在“同一条 case 需要按多组候选值独立跑多轮”时使用。

示例：

```yaml
变参:
  参数:
    - 设备类型
    - 采样率
  候选值:
    - [Neusen HEEG, 1000]
    - [Neusen U32, 2000]
```

当前结论：

1. 图形交互类 case 当前不推荐滥用 `变参`
2. 如果只是单条场景中的几个固定值，优先放进 `数据`

## 六、支持文件

### 目录级

- `init.yaml`
- `cleanup.yaml`

### 套件级

- `suite_setup.yaml`
- `suite_cleanup.yaml`

## 七、图形交互 case 补充规则

1. 图形交互比例统一写进 `数据` 段。
2. 比例通过独立比例工具辅助读取。
3. 拖动等非默认交互必须显式写 `动作`。
4. 图形交互类 case 仍然是正式步骤式 case，不是特殊第二格式。

### 历史回放剪辑类 case 额外约定

1. 推荐写法是“变量集中到 `数据` 段、步骤正文简化、默认点击省略、只保留关键断言”。
2. 不追求每一步都带独立断言，避免 case 退化成操作流水账。
3. 推荐只保留这类关键断言：
   - 历史回放窗口已出现
   - 数据剪辑窗口已出现
   - 数据剪辑片段已生成
   - 剪辑完成窗口已关闭
   - 历史回放窗口已关闭
   - 主窗口已新增最新剪辑记录
4. 历史回放剪辑类 case 当前不推荐使用 `变参`。

## 八、YAML 结构注意事项

1. 步骤之间允许空行。
2. 但每个步骤必须从独立的 `-` 开始。
3. 缩进层级必须保持一致。
4. 不允许破坏列表结构后再把问题误判成“空行报错”。
