# 需求说明

## 项目目标

创建一个基于 Python 的 WPF 桌面 UI 自动化工程，并逐步沉淀为可复用、可组合、可编号、可持续扩展的模块化测试资产。

## 当前阶段目标

当前阶段重点不是做完整低代码平台，而是验证并固定以下主线能力：

- 能稳定驱动目标 WPF 软件完成真实业务操作
- 能以“正式用例 + 大模块 + 元素清单 + pytest 执行层”的方式组织测试资产
- 能在失败时自动保留截图、错误信息和结构化报告
- 能让测试同事通过编辑 YAML 完成 case 和大模块维护，而不是直接修改底层 Python 实现

## 当前范围

- 启动或连接指定 exe
- 用正式 YAML 用例定义完整测试场景
- 用模块链编排业务流程
- 在“新建患者”场景下验证大模块模型可行性
- 支持一个 case 串多个大模块
- 支持 `自动` / `复用已有应用` 两种会话策略
- 支持在 VS Code 中直接运行 Python 入口文件执行单个或多个 case
- 输出日志、失败截图、pytest HTML 报告、JSON 报告和 Word 报告
- 用 `pytest` 统一承接收集、筛选、执行和结果治理

## 当前主设计要求

- 编码语言固定为 Python
- UI 自动化基于 `pywinauto` + UIA
- 默认开发与运行工具切换为 VS Code
- 正式用例采用中文 YAML，并使用“模块调用式”结构
- 固定层级：动作、元素、大模块、完整用例、pytest 执行层
- 不再把“小模块”作为公开架构层
- 一个 case 可以串多个大模块
- 大模块采用“可编辑定义文件 + 稳定执行实现”的组合
- 自然语言动作、模块、元素、代码位置之间必须有固定映射关系
- 动作映射、元素清单、模块注册和映射文档属于长期资产，不做负向删改
- 报告默认输出成功摘要，故障单独展开
- 运行产物至少包含日志、失败截图、pytest HTML 报告、JSON 报告和 Word 报告

## 当前入口要求

- `run_demo.py`：用于单个默认 case 演示
- `run_case_suite.py`：用于在 VS Code 中直接运行并手动选择某几个或全部 case
- `pytest`：负责 case 展示、结构校验、标签筛选、UI 套件执行和结果治理

## 当前样板范围

当前已落地的正式样板包括：

- 大模块：`新建患者` -> `patient.create`
- 正向 case：`TC_PATIENT_001`
- 负向 case：`TC_PATIENT_002`

## 当前不包含

- 可视化拖拽平台
- 通用低代码流程编排器
- 复杂日期控件自动化
- 大规模业务场景全覆盖
- 成功截图批量入报告
- 完整 CI/CD 或远程调度体系
