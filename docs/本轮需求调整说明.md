## 3. 分项说明

### 3.1 用例参数与多轮执行

原需求一句话总结：单个 case 应尽量只保留一组核心变参，其余参数固定，并支持按变参候选值和循环次数展开多轮执行，每轮都要独立给出 Pass/Fail 结果。

当前系统调整情况：

- 正式 case 顶层已支持 `变参`、`循环次数`、`失败即停` 三个关键能力。
- 当前数据模型只允许一个 `变参` 块，这实际上已经把“单 case 只保留一组核心变参”的约束固化进了主线结构。
- `FormalCaseRunner` 会按照“先变参值、再循环轮次”的顺序构建执行计划，并为每一轮生成独立执行名称、参数快照和结果。
- 每轮执行结果不再只看“脚本是否跑完”，而是按模块断言结果归并成 `PASS / FAIL / INTERRUPTED / NOT_RUN`。
- 现有样板 `src/heeg_auto/cases/系统设置/设备设置/采样率校验.yaml` 和 `src/heeg_auto/cases/系统设置/设备设置/设备设置循环.yaml` 已分别覆盖多值展开与循环执行场景。

当前判断：该项主线能力已完成。

后续建议：

- 继续保持“一个 case 只保留一组核心变参”的约束，不建议回退到多组变参并存，否则会迅速放大 case 复杂度和报告阅读成本。
- 如后续确有组合参数测试需求，建议单独设计“矩阵执行”能力，而不是直接突破当前正式 case 结构。

### 3.2 YAML 编写规范与特殊输入兼容

原需求一句话总结：需要统一 YAML 参数写法，明确双引号使用规则，并保证带双引号或特殊字符的输入值能够被正确解析和传递。

当前系统调整情况：

- 项目已形成独立文档 `docs/YAML编写规范.md`，明确规定“默认参数不加双引号”。
- loader 会把 YAML 中的裸值统一规范为文本后再传给执行层，降低 IP、采样率、病历号等字段被 YAML 自动转型带来的歧义。
- 对“值本身包含双引号”的场景，当前规范要求使用单引号包裹整段文本，例如 `'"123"'`，并且已有治理测试覆盖该行为。
- loader 已增加重复键拦截，避免同一字段写多次后被 YAML 静默覆盖，这对参数治理是一次有效补强。
- 当前项目已经同步输出规范文档，不再依赖口头约定。

当前判断：该项已完成。

当前边界：

- 当前更偏“规则先行 + loader 基础保护”，还没有单独的 YAML lint 命令去检查书写风格是否完全符合规范。

后续建议：

- 如后续编写人员增多，建议增加一层静态校验，专门检查 case 是否违反 YAML 编写规范，而不是等到运行时发现问题。

### 3.3 框架组织、中文化与按目录执行

原需求一句话总结：框架需要按业务目录分层组织 case，支持中文命名，并能按目录批量执行和清晰展示层级关系。

当前系统调整情况：

- 当前正式 case 已按业务目录组织在 `src/heeg_auto/cases` 下，目录和 YAML 文件名均已中文化。
- 中文字段、中文模块名、中文目录、中文报告展示都已进入主线，不再局限于英文命名。
- `run_case.py` 已支持按文件夹名称批量选择 case，不需要逐条手工输入。
- `tests/support/case_catalog.py` 已能收集 case 的目录、模块链、执行计划标签，并在运行选择和 pytest 展示中输出。
- 当前目录结构已经具备继续扩展更多层级的基础，至少从仓库组织方式上满足“按业务/模块维度扩展”要求。

未完全满足的点：

- 当前层级展示仍以“目录路径 + 模块链 + case 名称”为主，已经能表达结构，但还没有专门的树状导航或更强的四层视图。
- 也就是说，当前系统已经具备“分层组织”和“可识别层级”，但“模块 / 子模块 / 用例文件 / 具体用例”的独立树形展示能力仍然偏轻量。

当前判断：该项部分完成。

后续建议：

- 如果技术领导后续更关注用例资产治理，可在 `run_case.py` 或 HTML 报告中增加显式树状视图，把“目录层级”和“case 文件层级”分开显示。
- 内部稳定标识仍建议保留 ASCII，例如模块 id，避免中文化继续下沉到执行内核后增加兼容成本。

### 3.4 测试环境管理、会话复用与环境隔离

原需求一句话总结：框架既要避免重复启动软件、支持多轮循环验证，也要提供公共初始化/清理和测试后环境恢复机制。

当前系统调整情况：

- 会话复用能力已落地。case 若不显式写 `启动软件` 模块，执行器会尝试直接连接已有应用会话，而不是重复启动软件。
- `run_case.py` 日常执行时默认不会在每条 case 后追加额外的关软件/开软件动作，这满足了“避免重复启动软件”的优化方向。
- 单 case 的 N 次循环执行、多值执行、失败即停和最终统计报告已经落地，可用于快速验证和稳定性复跑。
- 执行链中已加入 watchdog，当界面长时间无进展时会强制保护性关闭软件，避免整轮任务长时间卡死。

未完成或已调整取舍的点：

- “目录级公共初始化/清理”当前没有保留在主线里，且项目文档已明确说明该能力被移除。
- 当前移除的主要原因是：按目录批量执行时，统一 before/after 很容易与 case 自身的会话策略冲突，尤其会误伤“复用已有应用”的场景，造成误关软件、打断上下文或引入新的不确定性。
- 当前主线已经转向“case 显式声明是否启动软件 + 执行器按会话策略复用”的模式，因此不再适合继续保留隐式目录钩子。
- “每组测试结束后环境必然恢复”这一点目前也不能算完全完成。因为 `run_case.py` 默认按连续会话运行，框架更偏向执行效率优先，而不是每条 case 后强制恢复到绝对干净状态。

当前判断：该项部分完成，其中“避免重复启动”和“循环执行”已完成，“目录级公共清理”和“严格环境隔离”未完成。

后续建议：

- 如果后续必须补环境恢复，建议不要回到“目录隐式 before/after”模式，而是新增显式执行策略，例如：
  - `clean_session`：每条 case 前后强制清理
  - `reuse_session`：允许共享会话，提高执行效率
  - `suite_setup / suite_teardown`：只对明确声明的批次执行
- 只有把“执行效率”和“环境隔离”分成可选策略，这个问题才不会继续互相冲突。

### 3.5 测试报告改进

原需求一句话总结：HTML 报告需要提升可读性，并明确区分成功、失败、异常中断和未执行，同时给出定位问题所需的失败信息。

当前系统调整情况：

- 当前项目已改为输出单一 HTML 报告，便于日常汇报和归档。
- 报告已显式区分 `PASS / FAIL / INTERRUPTED / NOT_RUN` 四类状态，并用不同颜色标识。
- 报告会展示用例名称、目录、执行计划、执行参数、模块参数快照、失败原因/断言信息、截图产物链接。
- 对异常或失败轮次，报告会展开失败模块、失败步骤和错误摘要，而不再只显示一个笼统的 `Fail`。
- 成功步骤明细默认收敛，异常信息单独展开，符合技术汇报时“概览清楚、问题聚焦”的阅读方式。

当前判断：该项已完成。

后续建议：

- 后续如需给更高层领导汇报，可在现有 HTML 报告之上再叠加一层“管理摘要页”，但不建议破坏当前面向技术定位问题的详细结构。

### 3.6 系统级路径管理

原需求一句话总结：系统类用例中的路径不应写死在 case 中，应统一从系统级配置中管理。

当前系统调整情况：

- 当前已把启动软件路径从 case 中抽离，统一由 `src/heeg_auto/config/app_config.yaml` 管理。
- 正式 case 中已经不再直接写 `软件路径` 或 `exe路径`，系统模块 `启动软件` 只关心 `会话模式`。
- 这次调整已经明显降低了环境切换和路径变更时修改 case 的成本。

尚未完全延展的点：

- 当前集中管理的路径主要是“启动软件路径”，还没有抽象成覆盖导入文件、导出目录、测试资源等更广泛场景的通用路径配置中心。
- 也就是说，这项需求在当前项目范围内已经部分落地，但更通用的系统路径治理框架还没有继续展开。

当前判断：该项部分完成。

后续建议：

- 如果后续会接入更多系统类 case，建议把路径配置扩展成“按环境切换”的系统配置，例如区分开发环境、测试环境、演示环境，并对外只暴露稳定别名。

### 3.7 断言判定机制建设

原需求一句话总结：需要建立统一断言判定列表，沉淀标准化 Pass/Fail 口径，并让 case 优先引用统一断言标准。

当前系统调整情况：

- 当前主线已经不再使用 `预期状态: PASS/FAIL`，而改为在模块层显式指定 `断言组`。
- 项目已新增 `src/heeg_auto/config/assertion_groups.yaml` 和 `docs/断言目录.md`，用于维护断言标准。
- 当前断言目录已覆盖断言名称、判定对象、判定条件、成功标准、失败标准、依据来源和备注说明，基本符合原需求提出的判定列表字段。
- `ModuleRunner` 会在执行时校验并使用对应断言组，说明该机制不是单纯文档，而是已经进入执行主线。

尚未完全满足的点：

- 当前断言标准主要覆盖已接入主线的模块，例如 `新建患者`、`设备设置`。
- `system.launch` 当前尚未建立显式断言组，仍以建立/复用会话为主。
- `设备设置 -> 设置失败` 这类失败断言目前保留了结构，但错误提示类断言还不够细。

当前判断：该项部分完成。

后续建议：

- 后续新增业务模块时，应把“补断言目录”视为接入主线的必备动作，而不是可选文档工作。
- 对已有失败类断言，建议继续补充更细的失败依据，例如具体错误提示文本、日志关键字或界面标识，提升评审时的可追溯性。

## 4. 综合结论

从当前项目状态看，这轮需求调整并不是“完全未做”，而是已经完成了主干框架的几项关键转向：

- case 从单次脚本执行转向了“单核心变参 + 多轮独立判定”的正式执行模型
- YAML 编写规范、中文化命名、断言组和统一报告已经进入主线
- 日常执行入口已经从 pytest 间接执行，收敛为更适合汇报和日常操作的 `run_case.py`

当前最值得向技术领导说明的未完成点，不是基础能力缺失，而是两类有意识的取舍：

- 目录级公共初始化/清理没有继续保留，因为它与当前“显式会话策略 + 连续执行提效”的主线冲突
- 环境隔离和断言标准化虽然已经起步，但还没有扩展成覆盖所有模块、所有批量场景的完整治理体系

如果下一阶段继续推进，建议优先做两件事：

1. 把“环境隔离”和“会话复用”拆成显式可选执行策略，避免继续互相制约。
2. 把断言目录和路径配置从“当前样板可用”推进到“新模块接入必须遵循”的工程约束。
