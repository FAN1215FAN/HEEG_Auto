# 需求说明

## 项目目标

创建一个基于 Python 的 WPF 桌面 UI 自动化工程，并逐步沉淀为可复用、可组合、可编号、可持续扩展的模块化测试资产。

## 当前阶段目标

当前阶段重点不是做完整低代码平台，而是验证并固定以下主线能力：

- 稳定驱动目标 WPF 软件完成真实业务操作
- 以“正式用例 + 大模块 + 元素清单 + pytest 执行层”的方式组织测试资产
- 在失败时自动保留截图、错误信息和结构化报告
- 让测试同事通过编辑 YAML 完成 case 和大模块维护，而不是频繁修改底层 Python
- 通过统一配置、断言组和编写规范降低用例维护成本

## 当前范围

- 启动或连接指定 exe
- 用正式 YAML 用例定义完整测试场景
- 用模块链编排业务流程
- 支持一个 case 串多个大模块
- 支持在 case 顶层定义一组核心变参，并按候选值独立展开执行
- 支持在 case 顶层配置循环次数，并在多轮执行时失败即停
- 用模块级断言组显式表达通过/失败判定
- 启动软件路径从 case 中抽离，统一由配置文件管理
- 输出日志、失败截图和自定义 HTML 报告
- 用 `run_case_suite.py` 承接日常正式 case 执行与汇报报告，用 `pytest` 承接治理测试与联调

## 当前主设计要求

- 编码语言固定为 Python
- UI 自动化基于 `pywinauto` + UIA
- 默认开发与运行工具为 VS Code
- 正式用例采用中文 YAML，并使用 `模块链` 结构
- 固定层级：动作、元素、大模块、完整用例、pytest 执行层
- 不再把“小模块”作为公开架构层
- 一个 case 可以串多个大模块
- 大模块采用“可编辑定义文件 + 稳定执行实现”的组合
- 启动软件作为显式系统模块写入 case 的 `模块链`
- 自然语言动作、模块、元素、代码位置之间必须有固定映射关系
- 动作映射、元素清单、模块注册、断言目录和映射文档属于长期资产，不做负向删改
- 参数默认不加双引号；当值本身需要输入双引号时，使用单引号包裹整段文本
- 报告默认输出执行概览、开始/结束时间、执行摘要、模块参数快照、故障截图链接；成功步骤不逐条展开，故障单独展开

## 当前入口要求

- `run_demo.py`：用于单个默认 case 演示
- `run_case_suite.py`：用于在 VS Code 中直接运行并手动选择某几个或全部 case
- `pytest`：负责结构校验、治理测试和可选 UI smoke 联调

## 当前样板范围

当前已落地的正式样板包括：
- 系统模块：`启动软件` -> `system.launch`
- 大模块：`新建患者` -> `patient.create`
- 大模块：`设备设置` -> `device.settings`
- 启动样板：`启动软件_01`
- 正向 case：`患者管理_01`
- 负向 case：`患者管理_02`
- 设备设置 case：`设备设置_02`
- 变参样板：`设备设置_01`
