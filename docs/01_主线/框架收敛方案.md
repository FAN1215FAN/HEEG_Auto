# 框架收敛方案

## 文档定位

这份文档不是结果摘要，而是项目框架收敛的长期主文档。后续每次涉及框架方向、主流程、主目录、正式 case 结构、资产结构、执行入口的调整，都必须先看这份文档，再同步更新这份文档。

## 背景

项目最初同时承接了几类诉求：

- 让不懂开发的测试同事也能维护自动化并编写 case
- 保留 `pytest`，方便研发提交后接入回归和 CI
- 面对数千到上万条手工用例时，自动化框架不能因为表达方式太重而失控
- 需求持续变动，评审中经常追加想法，框架不能频繁推倒重来
- 项目对象是 WPF 上位机软件，而且大部分场景必须走桌面 UI

围绕这些约束，项目必须收敛成“低门槛编写入口 + 代码执行内核”的混合架构，而不是纯 Python 脚本，也不是纯录制式 RPA 流程。

## 已达成协议

以下内容来自当前阶段已经明确的共识，后续默认继续遵守：

1. 正式执行内核继续使用 `Python + pywinauto + UIA + pytest`，不再切换主框架。
2. 研发团队可以配合补稳定 `AutomationId`，并配合处理 WPF 自定义控件的自动化暴露能力。
3. 项目不再保留双轨主线，外部不再区分 `V1 / V2`，正式方案只有当前步骤式单轨。
4. 正式 case 真源统一为中文步骤式 YAML。
5. `run_case.py` 面向测试日常执行，`pytest` 面向研发、治理和 CI，两者共用同一份 case 真源。
6. 断言必须继续资产化，不能散落成大量不可维护的底层代码调用。
7. 环境初始化与会话复用策略是独立议题，但不再反向影响主框架收敛。
8. 文档必须持续更新，不能再用“新建一份摘要”覆盖原有长期文档。

## 最终框架结论

### 1. 执行内核

- 语言：Python
- UI 自动化底座：`pywinauto` + UIA
- 测试治理与 CI 入口：`pytest`
- 正式运行入口：`run_case.py`

### 2. case 真源

- 正式 case 只采用中文步骤式 YAML
- 正式 case 目录只认 `src/heeg_auto/cases/`
- 正式 case 不再依赖旧模块链目录、旧元素目录、旧兼容目录

### 3. 资产真源

- 窗口资产：`src/heeg_auto/assets/windows/`
- 元素资产：`src/heeg_auto/assets/elements/`
- 断言资产：`src/heeg_auto/assets/assertions/`
- 资产统一由 `src/heeg_auto/assets/asset_store.py` 读取

### 4. 编写入口

当前阶段先接受 YAML 作为正式编写入口。后续如果补测试友好的上层录入工具，也必须输出为正式步骤式 YAML，而不是再引入第二套正式执行格式。

### 5. 断言治理方向

断言分三层治理：

- 基础断言动作层：执行器直接支持的通用断言能力
- 命名断言资产层：测试同事通过断言名称引用已有断言
- 领域断言模板层：针对脑电、历史回放、图形校验等高频场景沉淀参数化模板

测试侧的目标不是手写底层断言逻辑，而是“选模板 + 填参数 + 审核 AI 草稿”。
