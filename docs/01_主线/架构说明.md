# 架构说明

## 文档定位

这份文档说明当前正式框架的分层、主执行链路、资产定位和环境支持方式。它只描述当前正式架构，不再描述已经取消的模块链、子模块链和版本双轨。

## 一、总体结构

当前正式架构可以概括为两层：

1. 执行层
   - `src/heeg_auto/config`
   - `src/heeg_auto/core`
   - `src/heeg_auto/pages`
   - `src/heeg_auto/runner`
2. 资产层
   - `src/heeg_auto/cases`
   - `src/heeg_auto/assets`

项目的正式业务主线统一为“步骤式 YAML case + 资产仓库 + pytest 治理”。

## 二、正式主链路

### 1. 入口层

- `run_case.py`
  - 面向测试同事的人工执行入口
- `pytest`
  - 面向研发、治理和 CI 的统一执行入口

### 2. case 收集与调度层

- `tests/support/case_catalog.py`
  - 扫描正式 case
  - 构建 pytest 展示名称
  - 支持按目录、编号、文件筛选
- `src/heeg_auto/runner/formal_suite_service.py`
  - 负责套件级执行
  - 协调目录生命周期、逐 case 执行和结果汇总

### 3. 生命周期层

- `src/heeg_auto/runner/directory_lifecycle.py`
  - 管理 `suite_setup.yaml / suite_cleanup.yaml`
  - 管理目录级 `init.yaml / cleanup.yaml`
  - 支持 `reuse_per_suite` 与 `reset_per_directory`

### 4. case 加载与执行层

- `src/heeg_auto/runner/formal_case_runner.py`
  - 承接单 case 运行
  - 挂接看门狗、失败工件留存、环境检查
- `src/heeg_auto/runner/step_case_loader.py`
  - 解析步骤式 YAML
  - 标准化字段、构建上下文、处理变量和变参
- `src/heeg_auto/runner/step_case_executor.py`
  - 把步骤转换为标准动作调用
  - 负责表单步骤、显式动作、步骤断言和执行计划

### 5. 资产解析层

- `src/heeg_auto/assets/asset_store.py`
  - 统一加载窗口、元素、断言资产
  - 负责按窗口和名称解析引用

### 6. UI 执行层

- `src/heeg_auto/core/actions.py`
  - 封装点击、输入、拖动、断言等动作
- `src/heeg_auto/core/driver.py`
  - 驱动真实 WPF 应用
- `src/heeg_auto/core/base_page.py`
  - 封装基础定位与页面辅助逻辑
- `src/heeg_auto/pages/`
  - 放置复杂窗口或页面专用能力

### 7. 报告与工件层

- `src/heeg_auto/core/reporting.py`
  - 汇总结构化结果
- `artifacts/`
  - 存放日志、截图、HTML 报告、检查工具输出等产物

## 三、正式资产结构

### 1. 窗口资产

- 目录：`src/heeg_auto/assets/windows/`
- 作用：
  - 提供稳定窗口名称
  - 提供窗口定位信息
  - 提供图形交互所需的窗口交互标定

### 2. 元素资产

- 目录：`src/heeg_auto/assets/elements/`
- 作用：
  - 提供控件定位信息
  - 提供控件中文业务名称
  - 为默认动作推断和表单填写提供基础

### 3. 断言资产

- 目录：`src/heeg_auto/assets/assertions/`
- 作用：
  - 将复杂结果判断收敛为命名断言
  - 让 case 只写断言名，不直接堆底层检查细节

## 四、正式 case 结构

正式 case 统一使用中文步骤式 YAML。顶层当前支持：

- `用例编号`
- `用例名称`
- `标签`
- `数据`
- `变参`
- `循环次数`
- `失败即停`
- `步骤`

单个步骤当前支持：

- `名称`
- `窗口`
- `元素`
- `按钮`
- `动作`
- `值`
- `文本`
- `超时`
- `参数`
- `断言`
- `可选`

## 五、执行模式

### 1. 推荐模式：`reuse_per_suite`

- 套件内尽量复用同一应用会话
- 切换目录时运行目录钩子，但不强制每条 case 后重启应用
- 更适合批量回归

### 2. 严格模式：`reset_per_directory`

- 切换目录后强制重建环境
- 更适合强隔离场景
- 执行成本更高

### 3. 失败恢复

即使处于复用模式，失败后仍允许执行硬重建恢复，避免污染后续 case。

## 六、图形交互架构位置

图形交互不是单独的第二套框架，而是当前步骤式执行层的一部分：

- 图形交互参数写在正式 case 的 `数据` 段中
- 图形交互动作仍由 `StepCaseExecutor` 调度
- 交互比例和区域标定统一放在窗口资产的 `交互标定` 中
- 辅助读取比例的工具位于 `run_ratio_picker.py` 与 `tools/inspectors/`

## 七、当前设计约束

1. 不再引入双轨正式框架。
2. 不再使用旧模块链作为正式业务表达。
3. 正式资产只维护在 `src/heeg_auto/assets/`。
4. 正式 case 只维护在 `src/heeg_auto/cases/`。
5. 复杂断言优先沉淀到断言资产，不直接在 case 中展开。
