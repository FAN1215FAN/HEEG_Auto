# 项目书

## 一、项目定位

本项目是公司 WPF 上位机软件的自动化测试工程，不是一次性脚本集合，也不是当前阶段就要交付的通用低代码平台。它的核心任务是先把一条长期可维护的桌面 UI 自动化主线做稳，再逐步降低测试同事的维护门槛。

## 二、为什么要做这个项目

当前业务软件场景复杂，很多关键流程必须通过桌面 UI 验证。与此同时，手工用例量级很大，需求变化频繁，如果每次都靠人工回归或让研发临时写脚本，成本会持续失控。

项目要解决的不是单一场景自动化，而是三件事同时成立：

1. 测试同事能维护正式 case。
2. 研发能把正式 case 接进 `pytest` 和 CI。
3. 自动化资产能随着需求增长持续沉淀，而不是每次重构推翻。

## 三、当前阶段目标

### 1. 工程主目标

- 固化正式执行内核
- 固化正式 case 结构
- 固化正式资产结构
- 固化治理入口和日常运行入口
- 形成稳定扩展方式

### 2. 业务主目标

- 优先覆盖核心 UI 场景
- 支持测试维护几百条量级的正式 case
- 支持参数化和重复场景复用
- 为图形、波形、历史回放等复杂判断场景预留断言治理能力

## 五、为什么不是纯 RPA 路线

测试同事确实更容易接受影刀或 PAD 一类流程式思维，但如果把正式真源完全建立在 RPA 流程图上，会有几个明显问题：

1. 大体量 case 的参数复用和批量治理会越来越困难。
2. 研发很难把流程图直接纳入 `pytest` 和代码评审体系。
3. 复杂断言、图形判断和大批量回归更适合落在代码执行内核上。

所以当前项目采取的是折中方案：

- 正式真源仍是 YAML
- 后续可以给测试补更友好的录入入口
- 录入入口的输出仍必须回到正式 YAML

## 六、为什么研发必须配合 `AutomationId`

WPF 自动化能否稳定，不取决于某个“神奇工具”，而取决于界面是否稳定暴露自动化属性。研发配合补稳定 `AutomationId` 后：

- 元素定位更稳定
- UI 变动后的维护成本更低
- 断言和操作都更容易做成可复用资产

对于自定义控件，仅靠外部工具硬猜通常不够稳定，还需要研发补控件的自动化暴露能力。

## 七、什么是 `AutomationPeer`

`AutomationPeer` 可以理解成 WPF 自定义控件对自动化世界的“说明书”。如果一个控件是自绘的、组合的、默认不暴露标准自动化属性，那么自动化工具很难知道它是什么、能不能点、当前值是什么。

研发补上后，自动化工具更容易拿到：

- 控件身份
- 控件类型
- 可点击、可选择、可读取状态
- 稳定的自动化树节点

对你们这种存在图形区、波形区、自定义界面的软件，这项配合会非常有价值。

## 八、当前项目组织方式

### 1. 测试同事负责

- 维护正式 case YAML
- 维护业务参数
- 复用已有资产与断言模板
- 审核 AI 生成的断言草稿和步骤草稿

### 2. 研发同事负责

- 维护执行内核和底层能力
- 配合补 `AutomationId`
- 为关键自定义控件提供自动化暴露能力
- 维护 `pytest` 治理链路

### 3. 共同负责

- 明确哪些场景必须自动化
- 明确哪些断言是领域级模板
- 及时把评审结论沉淀到文档

## 十一、当前最大风险

1. 复杂断言如果不尽快模板化，后续测试维护会卡死。
2. 旧模板和旧示例如果不及时清理，会持续误导接手同事。
3. 文档如果不持续更新，项目会再次出现“代码往前走、文档留在过去”的情况。

## 十二、后续三条硬规则

1. 不再频繁更换核心框架。
2. 不再长期保留旧模块时代正式资料。
3. 每次变动都必须同步更新长期文档。
