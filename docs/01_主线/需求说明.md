# 需求说明

## 文档定位

这份文档记录项目的业务目标、工程目标、边界、约束和验收方向。它不是一次性立项说明，而是会跟着项目演进持续更新的需求主文档。

## 一、项目背景

本项目是面向公司 WPF 上位机软件的自动化测试工程。项目存在三个现实背景：

1. 软件业务复杂，场景大量依赖桌面 UI，几乎不能靠接口测试替代。
2. 手工用例总体规模很大，原始量级可以达到数千到上万条，但真正适合沉淀成自动化 case 的规模预计为几百条。
3. 需求和评审意见经常变化，项目必须能在变动中继续推进，而不是每次评审后推倒重来。

## 二、项目目标

### 1. 业务目标

- 让自动化能够真实覆盖 WPF 上位机的核心业务场景
- 让测试同事可以维护自动化 case，而不是所有变更都依赖研发改代码
- 让研发在代码提交后可以通过 `pytest` 执行自动化回归
- 让复杂业务流程、重复参数组合和大批量回归具备可扩展的表达方式

### 2. 工程目标

- 固化一套长期可维护的正式框架
- 把窗口、元素、断言沉淀为资产
- 把正式 case 收敛成统一真源
- 在失败时保留日志、截图、报告等证据
- 让后续功能扩展主要表现为“补资产、补步骤、补断言”，而不是反复改框架

## 三、领导意图与团队共识

根据当前阶段已经确认的方向，项目必须同时满足两条线：

1. 测试线：非开发同事也应能维护 case，至少能以 YAML 或更友好的上层录入方式完成维护。
2. 研发线：必须保留 `pytest` 作为治理、回归和 CI 入口。

因此项目最终不能是纯 RPA，也不能是纯代码脚本，而必须是：

- 上层可低门槛编写
- 下层可代码治理与 CI
- 中间有稳定的正式 case 真源和正式资产真源

## 四、当前正式范围

### 1. 正式执行内核

- Python
- `pywinauto`
- UIA
- `pytest`

### 2. 正式 case 范围

- 中文步骤式 YAML
- 目录位于 `src/heeg_auto/cases/`
- 由 `run_case.py` 和 `pytest` 共用

### 3. 正式资产范围

- 窗口资产
- 元素资产
- 断言资产
- 统一位于 `src/heeg_auto/assets/`

### 4. 正式报告范围

- 执行日志
- 失败截图
- 结构化结果
- pytest 报告

## 五、当前关键约束

### 1. UI 场景约束

- 大部分业务必须走桌面 UI，不能寄希望于用接口测试替代主体业务
- WPF 自定义控件、自绘图形、复杂窗口切换是长期难点

### 2. 维护成本约束

- case 量上来之后，不能让测试同事理解大量底层 Python 断言和定位代码
- 断言数量会快速增长，必须做成可选模板和命名资产

### 3. 研发配合约束

- 研发需要配合补稳定 `AutomationId`
- 对关键自定义控件，需要配合补 `AutomationPeer` 或至少暴露稳定自动化属性

### 4. 过程管理约束

- 项目没有成型的前置需求文档，很多信息需要从评审与实际开发中持续沉淀
- 因此文档不能写一次就算完，必须持续更新

## 六、当前最难的需求点

### 1. 图形和波形相关断言

脑电业务里最难的不是点击或输入，而是判断：

- 波形是否正确
- 图形是否正确
- 某些关键值是否正确
- 历史回放和数据剪辑后的结果是否符合预期

这意味着断言体系必须是重点建设对象，而且大概率需要 AI 参与断言草稿生成，但最终仍要人工审核入库。

### 2. case 编写友好性

测试同事更容易接受类似影刀或 PAD 的流程式思维，但项目正式真源仍需保持为 YAML，才能适应批量维护、参数复用、代码评审和 CI。

### 3. 需求频繁变动

项目必须允许新增动作、新增参数、新增断言模板、新增资产文件，但不能因为这些变化不断重写主框架。

## 七、正式技术方案要求

1. 不更换主执行内核。
2. 不再保留双轨正式框架。
3. 正式 case 真源只保留步骤式 YAML。
4. 正式资产真源只保留 `assets/windows`、`assets/elements`、`assets/assertions`。
5. 断言必须走资产化、模板化路线。
6. 环境初始化与清理策略允许单独迭代，但不能影响正式 case 结构。
7. 所有结构调整都必须同步更新文档。

## 八、当前不做的事情

1. 当前不把项目做成完整低代码拖拽平台。
2. 当前不把影刀/PAD 作为唯一正式真源。
3. 当前不把所有手工用例一比一转成自动化。
4. 当前不为了兼容历史结构而长期保留旧模块时代文档和代码。

## 九、验收判断

当前阶段判断项目是否走在正确方向上，主要看以下几点：

1. 新业务是否能通过补资产和补步骤快速落地。
2. 测试是否能在不改底层 Python 的前提下维护大部分 case。
3. 研发是否能通过 `pytest` 跑通正式回归。
4. 断言是否开始从零散写法收敛成命名资产和模板。
5. 文档是否能真实反映项目状态，而不是停留在过时说明。

## 十、本轮已确认调整

### 框架层

1. 正式方案继续使用 `Python + pywinauto + UIA + pytest`，不再更换主执行内核。
2. 正式 case 只保留中文步骤式 YAML。
3. 对外不再区分 `V1 / V2`。
4. 正式资产只保留 `windows / elements / assertions` 三类目录。

### case 编写层

1. 测试当前阶段接受 YAML 作为正式维护格式。
2. 历史回放剪辑类 case 采用“`数据` 段集中变量、步骤正文简化”的写法。
3. `变参` 只在真正需要多轮独立执行时使用。
4. `名称` 不是技术强制项，但继续建议保留。

### 断言层

1. 断言必须继续资产化。
2. 测试侧后续目标是“选命名断言 / 模板断言”，而不是理解底层断言代码。
3. 图形和波形相关断言列为后续重点建设对象。

### 文档层

1. 文档必须持续更新，不能再压缩成结果摘要。
2. 主文档后续采用增量更新，不再走“重写一版替换旧版”的方式。
