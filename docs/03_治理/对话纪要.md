# 对话纪要

## 文档定位

这份文档只记录关键决策、协议和阶段结果，不做全文聊天回放。每次涉及框架方向、正式口径、主文档恢复、重要工程决策时，都必须同步更新。

## 时间线

### 2026-04-14：框架方向定稿

#### 已达成结论

1. 项目继续使用 `Python + pywinauto + UIA + pytest` 作为执行内核。
2. 正式 case 真源统一为中文步骤式 YAML。
3. `run_case.py` 面向测试，`pytest` 面向研发和 CI。
4. 不再长期保留双轨正式框架。
5. 正式资产统一进入 `src/heeg_auto/assets/`。

### 2026-04-14：WPF 自动化能力边界确认

#### 已达成结论

1. 研发团队可以配合补稳定 `AutomationId`。
2. 对关键自定义控件，研发理论上可以配合补自动化暴露能力。
3. 项目大部分场景必须走桌面 UI。
4. 正式自动化 case 规模预计为几百条，而不是把上万条手工用例一比一搬进来。

### 2026-04-14：测试维护目标确认

#### 已达成结论

1. 测试同事长期目标是通过更友好的上层录入方式维护 case。
2. 当前阶段接受 YAML 作为正式维护格式。
3. 断言是当前测试维护最大痛点，后续必须做成命名断言和模板化断言。

### 2026-04-14：代码主线瘦身确认

#### 已达成结论

1. 旧模块链、旧元素目录、旧模块目录、旧兼容目录不再是正式主线。
2. 项目中不再保留 `V1 / V2` 作为正式命名。
3. 环境策略独立评审，不与命名收敛绑死。

### 2026-04-14：环境策略能力确认

#### 已达成结论

1. 基础能力支持 `reuse_per_suite`。
2. 基础能力支持 `reset_per_directory`。
3. `suite_setup.yaml / suite_cleanup.yaml` 已纳入正式支持能力。
4. 具体业务采用哪套策略，后续单独定稿。

### 2026-04-14：文档恢复决定

#### 已达成结论

1. 前一轮文档被错误压缩成结果摘要，必须恢复。
2. 项目书、框架收敛方案、瘦身迁移清单、需求说明、资产总表、资产缺口清单都必须恢复成长期维护文档。
3. `docs/模板` 和 `docs/示例` 中的旧模块内容必须清理或归档。

### 2026-04-14：历史回放剪辑专题结论补录

#### 已达成结论

1. 历史回放剪辑类 case 应采用“变量集中到 `数据` 段、步骤正文简化、默认动作省略、关键断言保留”的写法。
2. 图形交互比例继续人工录入，但必须通过独立比例工具降低录入成本。
3. `名称` 字段不是强制字段，但强烈建议保留。
4. `变参` 只在“候选值每条都需要独立跑一遍 case”时使用，不能滥用。
5. 该专题已经沉淀出一套可复用的写法、比例工具和专题说明，不再只是单条 case 跑通。

### 2026-04-14：文档同步与残留收口规则确认

#### 已达成结论

1. 文档体系已完成主体恢复，后续进入持续维护阶段。
2. 每次结构、脚本、正式口径或迁移状态调整，都必须同步更新主文档进度。
3. `v2` 标记和历史导入脚本命名不再继续保留在正式面。
4. README、CHANGELOG、对话纪要和受影响主文档继续作为默认同步集合。

### 2026-04-14：正式 case 目录归属确认

#### 已达成结论

1. 正式 case 目录继续按业务域分组维护。
2. `历史回放剪辑` 不再归属 `患者检查管理/`，正式位置调整为 `回放数据/历史回放剪辑/`。
3. `回放数据 / 实时录制 / 患者检查管理 / 系统设置 / ERP回放 / 公共功能` 是当前软件设想的大分类区，后续允许继续增加新的同级分类。
4. 每个大分类区下允许继续拆分下级文件夹，但当前不强制定统一拆分原则。
5. 下级目录后续可能参考现有手工用例分类，也可能参考大功能下的窗口或功能结构；在定稿前，允许测试按实际需要先新增文件夹。
6. 当前下一步优先级调整为“目录边界收口 -> 断言入口治理 -> 环境策略定稿”。

### 2026-04-15：UI 异常等待 fail-fast 调整

#### 已达成结论

1. 正式执行链中 `AssetStore()` 的默认根目录必须直接指向 `src/heeg_auto/assets/`，不能再依赖额外传参。
2. UI 无进展 watchdog 默认值收紧到 `20` 秒，异常等待不再默认挂几分钟。
3. 启动阶段只要主窗口已经出现，就不再长时间阻塞在 `ready` 或 `set_focus`；相关失败改为告警优先。
4. 套件执行时继续保持“当前 case 快速失败并记录异常；若还有其他 case，则继续执行后续 case”的口径。

### 2026-04-17：Jenkins 串联联调结论补录

#### 已达成结论

1. 当前项目已经作为安装完成后的第二阶段自动化测试接入 Jenkins 串联批处理。
2. Jenkins 当前联调入口可收敛为 `python -m pytest --case-id 历史回放剪辑_01 -s`。
3. Jenkins 不需要额外添加“启动软件”命令，软件启动由目录初始化和正式执行链负责。
4. 当前 Jenkins 串联失败表现与单独运行本项目时一致，已收敛为本项目自身的启动识别问题，而不是安装项目串联命令问题。
5. 当前最新现场现象是：软件可以被拉起，但 watchdog 在启动步骤约 20 秒后判定无进展，随后在 30 秒内未识别到主窗口。

### 2026-04-20：历史回放剪辑时长字段收敛

#### 已达成结论

1. 历史回放剪辑类 case 中不再要求测试手工填写 `剪辑最小时长 / 剪辑最大时长 / 主窗口新增记录时长`。
2. `剪辑开始时间 / 剪辑结束时间` 继续作为这类 case 的核心业务时间字段，由执行层统一自动推导时长范围。
3. 数据剪辑窗口中的片段断言与主窗口最终新增记录断言统一改成范围判断，默认容差先收敛为 `±1` 秒。
4. `run_ratio_picker.py` 继续保持为独立调试工具，不直接插入正式执行链；降低跨机器比例调参成本和收敛断言规则是两条并行治理线。
5. 这次专题调整同时暴露出“代码变更必须同步文档”的治理要求需要严格执行，后续继续按受影响文档逐份追加日期化更新说明。

### 2026-04-20：首次提交清理口径补录

#### 已达成结论

1. 首次推送 GitHub 前需要优先清理本机化文件，避免把 VS Code 本地解释器配置、pytest 缓存目录和临时测试文件直接提交。
2. 这类清理优先通过 `.gitignore` 收口，而不是依赖提交前手工逐个排除。
3. 首次提交清理同样属于“有意义的改动”，需要按治理要求同步更新变更记录和主文档。
