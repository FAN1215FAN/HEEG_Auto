# 维护流程

## 文档定位

这份文档说明后续改项目时的默认维护顺序、文档同步规则，以及 `CHANGELOG` 记录规则。

## 一、当前维护原则

1. 新增正式 case 时，默认使用步骤式结构。
2. 新增 case 之前先补窗口、元素、断言资产。
3. 复杂断言优先沉淀成命名断言资产，不在 case 中堆底层判断。
4. 所有结构性改动默认遵循 [框架收敛方案](../01_主线/框架收敛方案.md)。
5. 环境初始化策略单独评审，不在日常需求里顺手漂移。
6. 默认环境模式为 `reuse_per_suite`，有明确需要时再切到 `reset_per_directory`。

## 二、推荐顺序

1. 先确认自动化标识和资产边界
2. 先补窗口、元素、断言资产
3. 再写 case
4. 再补测试
5. 同步文档和变更记录
6. 提交前再复核一次进度状态、未完成事项和最近更新日期

## 三、文档同步硬要求

每次有意义的改动，至少同步检查：

- `README.md`
- `CHANGELOG.md`
- `docs/03_治理/对话纪要.md`
- `docs/01_主线/架构说明.md`
- `docs/01_主线/目录说明.md`
- `docs/01_主线/运行指南.md`
- `docs/03_治理/维护流程.md`

如果改动涉及资产，还必须同步：

- `docs/02_资产规范/资产总表.md`
- `docs/02_资产规范/资产缺口清单.md`
- 相关资产规范文档

如果改动涉及框架方向，还必须同步：

- `docs/01_主线/框架收敛方案.md`
- `docs/01_主线/瘦身迁移清单.md`
- `docs/01_主线/需求说明.md`

如果改动影响阶段进度、收敛状态或迁移判断，还必须同步：

- 当前状态
- 已完成项 / 未完成项
- `最近更新`

## 四、文档更新规则

1. 后续所有长期文档都采用持续更新，不再做压缩式重写。
2. 每次更新必须在对应文档底部追加“最近更新”日期记录。
3. 如果一次改动同时影响多份文档，每份受影响文档都要各自补日期记录。
4. 旧内容原则上保留，通过追加更新说明体现变化，不再把旧内容整体删薄。
5. 每次结构、脚本、正式口径或迁移状态调整，都要同步更新对应主文档的进度字段，不能只改代码或只改结论。

## 五、CHANGELOG 记录策略

### 必须记入 CHANGELOG 的变更

1. 正式框架方向变化
2. 正式目录结构变化
3. 正式运行入口变化
4. 正式资产结构变化
5. 环境模式与生命周期能力变化
6. 重要专题能力新增，例如图形交互能力、比例工具、专题断言能力
7. 会影响项目可维护性的文档治理调整

### 通常不单独记入 CHANGELOG 的变更

1. 纯错别字修正
2. 不影响口径的小型文案润色
3. 细小测试断言文字调整

## 六、当前禁忌

1. 不再把长期主文档压缩成结果摘要。
2. 不再把旧模块时代资料继续留在正式面。
3. 不再动了代码却不更新文档。
