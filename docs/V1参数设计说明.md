# V1 参数设计说明

## 1. 文档目标

本文用于固化 HEEG Auto 当前版本的最小 pytest 参数口径。

本项目使用 pytest 的核心原因不是让测试人员直接手敲大量命令，而是给自动部署/自动回归机器提供一个稳定、标准、可无人值守调用的 CLI 执行入口。

因此，第一版设计强调三点：

- `cases/` 继续作为唯一正式 case 资产来源
- `pytest` 只暴露最小可用参数集，避免维护面过大
- 保留后续扩充能力，但当前不把复杂参数纳入项目日常支持范围

## 2. 当前执行分工

### 2.1 case 编写

- 测试人员继续在 VS Code 中维护 `src/heeg_auto/cases/` 下的 YAML
- case 编写逻辑不因引入 pytest 而改变
- `init.yaml` / `cleanup.yaml` 仍属于同一套 case 资产体系中的目录支持文件

### 2.2 人工执行入口

- 人工选择执行入口为 `run_case.py`
- 该入口用于本地手动选择某个 case、多个 case 或某个业务目录执行

### 2.3 无人值守执行入口

- 无人值守入口为 `pytest`
- 主要给自动部署工具、回归机、后续 CI/定时任务调用

### 2.4 tests 目录职责

`tests/` 不是 `cases/` 的镜像目录，不要求每个 YAML case 都生成一个对应的 `test_xxx.py`。

`tests/` 当前只承担两类职责：

- pytest 桥接入口：把 `cases/` 中的 YAML 正式 case 暴露给 pytest 收集和执行
- 框架治理测试：校验 loader、catalog、报告、目录生命周期等底层能力是否正常

## 3. 最小参数原则

当前项目正式支持的参数只分两类：

### 3.1 推荐的 pytest 原生参数

- `-h`：查看帮助
- `-s`：显示执行过程输出

### 3.2 项目自定义参数

- `--run-formal`
- `--run-ui`
- `--case-dir`
- `--case-file`
- `--case-id`
- `--stall-timeout`

使用 `--case-dir`、`--case-file`、`--case-id` 时，不再要求额外补 `--run-formal`。

除上述参数外，其余 pytest 原生参数默认视为“pytest 自身能力”，不是当前项目对测试团队承诺维护的主参数面。

## 4. V1 参数设计表

| 参数 | 类型 | 是否当前主支持 | 作用 | 示例 |
| --- | --- | --- | --- | --- |
| `-h` | pytest 原生 | 是 | 显示帮助 | `python -m pytest -h` |
| `-s` | pytest 原生 | 是 | 显示执行过程输出 | `python -m pytest -s` |
| `--run-formal` | 项目参数 | 是 | 只运行正式 YAML case 主线 | `python -m pytest --run-formal` |
| `--run-ui` | 项目参数 | 是 | 运行真实桌面 UI 自动化 | `python -m pytest --run-ui` |
| `--case-dir` | 项目参数 | 是 | 按业务目录筛选正式 case | `python -m pytest --case-dir 设备设置 -s` |
| `--case-file` | 项目参数 | 是 | 按单个正式 case 文件执行 | `python -m pytest --case-file src/heeg_auto/cases/系统设置/设备设置/采样率校验.yaml -s` |
| `--case-id` | 项目参数 | 是 | 按 case 编号筛选正式 case | `python -m pytest --case-id 患者管理_01 -s` |
| `--stall-timeout` | 项目参数 | 是 | 指定 UI 无进展超时秒数 | `python -m pytest --case-dir 设备设置 --stall-timeout 120 -s` |

## 5. 参数组合约束

### 5.1 case 筛选参数可直接触发正式 case 执行

以下筛选参数可直接触发正式 YAML case 执行：

- `--case-dir`
- `--case-file`
- `--case-id`

使用这些参数时，不再要求额外补 `--run-formal`。

### 5.2 `--case-file` 优先级最高

- 指定 `--case-file` 时，执行目标应收敛到该文件
- 不与 `--case-dir`、`--case-id` 同时使用

### 5.3 `--stall-timeout` 的作用范围

- `--stall-timeout` 仅用于真实 UI 路径
- 它只影响超时保护，不影响 case 编写方式

## 6. 推荐命令

### 6.1 查看帮助

```bash
python -m pytest -h
```

### 6.3 跑全部正式 case

```bash
python -m pytest --run-formal --run-ui -s
```

### 6.4 跑某个业务目录

```bash
python -m pytest --case-dir 设备设置 -s
```

### 6.5 跑单个 case 文件

```bash
python -m pytest --case-file src/heeg_auto/cases/系统设置/设备设置/采样率校验.yaml -s
```

### 6.6 跑某个 case 编号

```bash
python -m pytest --case-id 患者管理_01 -s
```

## 7. 关于报告与日志能力

收缩项目参数面，不会削弱 pytest 的报告与日志能力。

也就是说：

- 项目只对最小参数集做主支持
- pytest 自带的日志捕获、终端输出、HTML/JUnit 报告能力仍可继续使用
- 后续若自动部署工具需要接报告文件，仍可在此基础上扩展

## 8. 当前结论

当前版本的稳定口径如下：

- `cases/` 是唯一正式 case 源
- `tests/` 是“桥接入口 + 治理测试”，不是 case 镜像
- 人工入口使用 `run_case.py`
- 无人值守入口使用 `pytest`
- 当前项目只主支持 `-h`、`-s` 和 6 个项目参数
- 后续可以扩充，但不在本版主动引入更多参数面
